# Nick

Nick owns `/nick` and replaces raw player access to CMI nicknames with a safer picker. Players do not type color codes directly; they pick curated presets, cleanup tools, history entries, or favorites, and the plugin applies the final trusted nickname through CMI from console.

Before enabling this jar, remove or disable CMI's own `/nick` alias/command registration in CMI's alias configuration so this feature can own `/nick`.

## Player Flow

`/nick` opens a hardened GUI with a live preview. The player can browse public colors, gradients, seasonal styles, group styles, economy unlocks, milestone styles, moods, tools, history, favorites, top nicknames, and bio settings.

The main GUI includes a "Turn Off Nickname" button next to the current preview. It uses the same safe clear path as `/nick clear`, dispatching the configured CMI command `cmi nick off {player} -s` while keeping Nick history and favorites intact.

The bottom-left player head on the main GUI shows a quick account snapshot: real name, money when Vault economy is available, EXP level, current Nick preview, previous Nick snapshot, and whether a legacy nickname was captured. Clicking it closes the GUI and prints `/nick status`.

The History & Favorites page keeps the two concepts separate. History is a rolling list of previous Nick choices and only starts after there is a previous Nick choice to store; the first successful Nick save has no previous Nick yet. Favorites are permanent saved slots and are labeled separately.

The visible nickname text is strict: only letters, numbers, and underscores are accepted. Raw legacy colors, MiniMessage, hex input, semicolons, pipes, tellraw-style text, command chains, and formatting codes are removed or rejected. The default rule also requires the nickname to stay related to the player's real Minecraft name, so `MyName666` can become `MyName` or `Name`, but unrelated names such as random memes are blocked.

Style presets use trusted CMI color syntax generated by the plugin, such as `{#bde0fe}` and `{#bde0fe>}Name{#ffc8dd<}`. Players cannot type these markers themselves.

## Legacy Nickname Migration

When Nick first sees a player, it asks CMI for the player's existing CMI nickname and stores a one-time legacy snapshot in shared playerdata. This lets long-time players see what they had before the safer Nick rules took over.

Legacy restores are limited by `legacy.restore-uses`, defaulting to one restore. A player can restore the captured legacy nickname while uses remain, even if that old nickname would fail the new related-name or style rules. After the restore count reaches zero, the legacy entry remains visible but view-only.

Legacy nicknames are intentionally not full participants in the new perk system. They cannot be shared to chat or Discord, cannot be saved as permanent favorites, cannot be rated, and are excluded from `/nick top`. Any new `/nick set`, style, tool, favorite, or preset use moves the player back into the new controlled Nick workflow.

Admins with `onembcmi.nick.admin` see gray `id: ...` lines at the bottom of GUI item lore for preset ids, page ids, tool ids, favorites, history entries, and legacy actions. This makes it easier to match GUI choices to YAML config keys.

## Config Files

Nick writes commented files under:

```text
plugins/1MB-CMIAPI/Nick/config.yml
plugins/1MB-CMIAPI/Nick/rules.yml
plugins/1MB-CMIAPI/Nick/presets/global.yml
plugins/1MB-CMIAPI/Nick/presets/gradients.yml
plugins/1MB-CMIAPI/Nick/presets/unique.yml
plugins/1MB-CMIAPI/Nick/presets/groups.yml
plugins/1MB-CMIAPI/Nick/presets/economy.yml
plugins/1MB-CMIAPI/Nick/presets/milestones.yml
plugins/1MB-CMIAPI/Nick/presets/moods.yml
plugins/1MB-CMIAPI/Nick/stats.yml
plugins/1MB-CMIAPI/Nick/logs/nick.log
```

`config.yml` controls cooldowns, GUI filler material, CMI command templates, sharing, celebration effects, economy behavior, top-list settings, and command allow-lists.

Each cleanup/randomizer tool can be individually enabled or disabled in `config.yml`. Disabled tools are hidden from the Tools GUI, removed from tab completion, and blocked if someone tries to run them manually:

```yaml
tools:
  stripnumbers:
    enabled: true
  lowercase:
    enabled: true
  uppercase:
    enabled: false
  capitalize:
    enabled: true
  initials:
    enabled: true
  random:
    enabled: true
```

`rules.yml` controls nickname relatedness, duplicate-name checks, blocked staff/server terms, filter patterns, and staff alerts.

The `presets/` files define solid colors, gradients, seasonal styles, group styles, economy unlocks, milestones, and mood colors. Each preset can be enabled, disabled, permission-locked, date-locked, month-locked, bold/italic decorated, or given a money/EXP unlock cost.

Important legacy paths:

```yaml
legacy:
  enabled: true
  capture-on-join: true
  restore-uses: 1
  reset-if-disabled: true
```

`legacy.enabled: false` stops legacy restores. When `legacy.reset-if-disabled` is also true, Nick checks joining players and players opening `/nick`; if their current CMI nickname still matches the stored legacy nickname, Nick clears the CMI nickname, sets their legacy restore count to `0`, and sends the configured `commands.legacy-disabled-mail` CMI mail. New Nick-generated names are not touched.

## Data

Per-player Nick data is stored in the shared 1MB-CMIAPI playerdata file under the `nick` section. It tracks:

- current plain nickname, style id, and generated CMI formatted nickname
- captured legacy CMI nickname, restore count, capture time, and whether the current nickname is legacy
- last name/style/tool/share/rating timestamps
- unlocked paid styles
- opt-in global announcement state
- sanitized bio
- rolling nickname history
- permanent favorite slots
- anonymous rating map keyed by voter UUID

The separate `stats.yml` stores style use counts for trending/debug output. `logs/nick.log` records successful changes, failed changes, filter catches, staff-impersonation attempts, ratings, clears, and admin resets.

## Safety Notes

Nick never uses RCON. It only dispatches reviewed direct console commands and blocks command templates containing command chains, pipes, semicolons, tellraw, execute/function, op/deop, reload/stop, or unsafe angle brackets. Trusted CMI hex tokens generated by Nick are allowed through so gradient nicknames still apply.

Player input is sanitized before storage and before display. Bio text preserves `<3`, but removes formatting and unsafe command-looking text.

Duplicate-name checks compare against known Bukkit offline player names so players cannot impersonate another player with a close real-name nickname. The check is intentionally conservative and can be reviewed with `/nick admin inspect <player>`.

## CMI, CMILib, And Hooks

CMI and CMILib are required. Nick delegates final nickname changes through configurable CMI console commands:

```text
cmi nick {nick} {player} -s
cmi nick off {player} -s
cmi broadcast {player} picked a new nickname: {preview}
cmi mail send {player} Your old legacy nickname is no longer supported by the current /nick system. We cleared it so everyone uses the safer Nick picker. Use /nick to choose a new nickname.
cmi realname {player}
```

Vault is optional for money-cost style unlocks. PlaceholderAPI receives the Nick placeholders through the shared library. DiscordSRV can be used by configuring `share.discord-command` or a safe staff alert command. LuckPerms can grant group, event, milestone, and bypass permissions.

The default nickname announcement uses `cmi broadcast`, and Nick adds that prefix to the runtime allow-list when the configured announcement command still uses it. Other custom announcement commands still need a matching entry in `commands.allowed-prefixes`.

## Future Notes

Nameplate support is intentionally deferred. When the nickname flow is stable on live, investigate whether per-player nameplates can be changed without breaking glow, scoreboard, tab list, or CMI nameplate behavior. A future prefix/nameplate pass could add safe tools for staff-managed player prefixes and suffixes, plus player-selectable unlocked prefixes from groups, events, milestones, or seasonal rewards.

Chat bubble tinting is also deferred: a future pass can investigate matching chat bubble color to the player's active Nick style.
